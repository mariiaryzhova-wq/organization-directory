
# План тестування API для MVP1 — Проєкт "Каталог організацій"
## 1. Модуль "Organizations" (Організації)

### 1.1. GET /api/organizations
*Опис: Отримання списку всіх активних/публічних організацій.*
 * **Позитивні сценарії:**
   * **TC-01:** Запит від неавторизованого користувача (гостя). Очікуваний результат: 200 OK, повертається масив об'єктів організацій.
   * **TC-02:** Перевірка структури JSON-відповіді. Очікуваний результат: кожен об'єкт містить обов'язкові поля (id, name, description, category_id, status тощо).
   * **TC-03:** Перевірка пагінації та фільтрації (якщо закладено в MVP1, наприклад: ?page=1&limit=10). Очікуваний результат: 200 OK, повертається обмежена кількість записів.
 * **Негативні сценарії:**
   * **TC-04:** Симуляція помилки бази даних / сервера. Очікуваний результат: 500 Internal Server Error, інформативне повідомлення про помилку у форматі JSON.

### 1.2. GET /api/organizations/pending
*Опис: Отримання списку організацій, які очікують на модерацію (підтвердження).*
 * **Позитивні сценарії:**
   * **TC-05:** Запит з правами Модератора / Адміністратора. Очікуваний результат: 200 OK, повертається список організацій, у яких поле status має значення pending.
 * **Негативні сценарії:**
   * **TC-06:** Запит від імені гостя (без токена автентифікації). Очікуваний результат: 401 Unauthorized.
   * **TC-07:** Запит від звичайного користувача (без прав модератора). Очікуваний результат: 403 Forbidden (доступ заборонено).

### 1.3. GET /api/organizations/:id
*Опис: Отримання детальної інформації про конкретну організацію за її ID.*
 * **Позитивні сценарії:**
   * **TC-08:** Запит з існуючим та валідним id (наприклад, /api/organizations/5). Очікуваний результат: 200 OK, дані у відповіді відповідають конкретному запису в БД.
 * **Негативні сценарії:**
   * **TC-09:** Запит із неіснуючим id (наприклад, /api/organizations/99999). Очікуваний результат: 404 Not Found.
   * **TC-10:** Запит із некоректним форматом id (наприклад, строка замість числа або невалідний UUID: /api/organizations/abc). Очікуваний результат: 400 Bad Request.

### 1.4. POST /api/organizations
*Опис: Створення (подання заявки на додавання) нової організації.*
 * **Позитивні сценарії:**
   * **TC-11:** Відправка запиту з усіма обов'язковими та валідними полями в JSON-тілі (Body). Очікуваний результат: 201 Created, у відповіді повертається об'єкт створеної організації з присвоєним id. Статус організації за замовчуванням має бути pending.
 * **Негативні сценарії:**
   * **TC-12:** Відправка запиту з відсутніми обов'язковими полями (наприклад, без назви організації). Очікуваний результат: 400 Bad Request + повідомлення валідації (наприклад, "name is required").
   * **TC-13:** Валідація типів даних та довжини полів (наприклад, занадто довга назва, невалідний email або номер телефону). Очікуваний результат: 400 Bad Request.
   * **TC-14:** Спроба відправити порожнє тіло запиту ({}). Очікуваний результат: 400 Bad Request.

### 1.5. PUT /api/organizations/:id/status
*Опис: Зміна статусу організації (модерація: схвалення або відхилення заявки).*
 * **Позитивні сценарії:**
   * **TC-15:** Оновлення статусу модератором (передача в Body нового статусу, наприклад: {"status": "approved"}). Очікуваний результат: 200 OK або 204 No Content. Статус організації в БД змінюється на новий.
 * **Негативні сценарії:**
   * **TC-16:** Спроба зміни статусу неавторизованим користувачем або звичайним юзером. Очікуваний результат: 401 Unauthorized або 403 Forbidden.
   * **TC-17:** Передача невалідного значення статусу (наприклад: {"status": "active_random"}). Очікуваний результат: 400 Bad Request.
   * **TC-18:** Запит для неіснуючого id організації. Очікуваний результат: 404 Not Found.

### 1.6. POST /api/organizations/import
*Опис: Масовий імпорт організацій (наприклад, завантаження файлу CSV/JSON).*
 * **Позитивні сценарії:**
   * **TC-19:** Завантаження файлу правильного формату (CSV/JSON) з валідними даними організацій. Очікуваний результат: 200 OK (або 201 Created), повертається звіт про успішний імпорт (наприклад, кількість доданих рядків).
 * **Негативні сценарії:**
   * **TC-20:** Завантаження файлу непідтримуваного формату (наприклад, .txt, .xlsx або .pdf). Очікуваний результат: 400 Bad Request або 415 Unsupported Media Type.
   * **TC-21:** Завантаження файлу, де частина даних містить помилки (порушення структури). Очікуваний результат: 400 Bad Request із переліком помилкових рядків (або транзакція скасовується).

## 2. Модуль "Categories" (Категорії)

### 2.1. GET /api/categories
*Опис: Отримання списку всіх доступних категорій для фільтрації організацій.*
 * **Позитивні сценарії:**
   * **TC-22:** Запит від будь-якого користувача (публічний ендпоінт). Очікуваний результат: 200 OK, повертається масив об'єктів категорій (id, title, slug тощо).
   * **TC-23:** Перевірка сортування категорій (якщо закладено логіку, наприклад, за алфавітом).

### 2.2. GET /api/categories/:id/organizations
*Опис: Отримання списку організацій, які належать до конкретної категорії.*
 * **Позитивні сценарії:**
   * **TC-24:** Запит з існуючим id категорії, яка має прив'язані організації. Очікуваний результат: 200 OK, у відповіді повертаються тільки ті організації, які належать до цієї категорії.
   * **TC-25:** Запит для існуючої категорії, до якої ще не прив'язано жодної організації. Очікуваний результат: 200 OK, повертається порожній масив [].
 * **Негативні сценарії:**
   * **TC-26:** Запит з неіснуючим id категорії. Очікуваний результат: 404 Not Found.
   * **TC-27:** Передача невалідного формату id категорії (наприклад, /api/categories/xyz/organizations). Очікуваний результат: 400 Bad Request.
 
 ### 3. Загальні нефункціональні перевірки (Для всіх ендпоінтів)
 * **Формат відповіді:** Перевірка, що сервер завжди повертає заголовок Content-Type: application/json.
 * **Обробка методів (HTTP Methods):** Перевірка відправки невірного HTTP-методу (наприклад, POST на /api/categories). Очікуваний результат: 405 Method Not Allowed.
 * **Час відповіді (Performance):** Базова перевірка швидкості відповіді API (запити мають виконуватися у межах до 200–500 мс за нормальних умов).
