## 1. Резюме (Executive Summary)

Проведено повний цикл тестування backend-частини проєкту "Каталог організацій" відповідно до наданого вхідного "Плану тестування API для MVP1" та фактичного вихідного коду. Тестування охопило чотири рівні: статичний аналіз коду, модульне тестування ізольованих функцій, інтеграційне тестування взаємодії контролерів із репозиторіями через маршрути Express, та наскрізне (end-to-end) тестування HTTP API за допомогою Supertest.

Усі 108 автоматизованих тестів виконано успішно (Jest "PASS"), однак частина тестів свідомо написана таким чином, щоб зафіксувати **фактичну** поведінку системи там, де вона відрізняється від очікуваної за специфікацією — це стандартна практика регресійного тестування при виявленні дефектів: тест документує "як є", доки дефект не буде виправлено.

У результаті тестування виявлено:
- **2 критичні дефекти (Bugs)**, що блокують ключовий функціонал (масовий імпорт CSV та фільтрацію організацій за категорією);
- **5 прогалин (Gaps)** — відхилень фактичної реалізації від наданого плану тестування та специфікації, переважно пов'язаних із відсутністю модуля автентифікації/авторизації;
- **0 регресій** у вже коректно реалізованому функціоналі (CRUD організацій, базова валідація, обробка помилок 400/404).

**Висновок:** базовий функціонал перегляду, створення та модерації організацій (без імпорту та без фільтрації за категорією) є **придатним до використання**. Функціонал масового імпорту CSV та фільтрації організацій за категорією **непрацездатний і потребує негайного виправлення** перед релізом. Функціонал контролю доступу (ролі, модерація) **відсутній** і має бути реалізований до того, як API стане доступним поза довіреним середовищем.

## 3. Стратегія та методологія тестування

Тестування побудоване за принципом піраміди тестування з рухом від ізольованих модулів до повного HTTP-стеку:

```
        ┌─────────────────────┐
        │   API / E2E (30)     │   ← Supertest, повний HTTP-цикл через Express app
        ├─────────────────────┤
        │  Integration (34)     │   ← маршрут + контролер + репозиторій + фейкова Prisma
        ├─────────────────────┤
        │     Unit (44)          │   ← ізольовані функції з моками залежностей
        ├─────────────────────┤
        │  Static Analysis (10)   │   ← ESLint + ручний код-рев'ю
        └─────────────────────┘
```

### 3.1 Статичне тестування
Виконано за допомогою ESLint (конфігурація `eslint:recommended`) та ручного код-рев'ю, спрямованого на пошук невідповідностей контрактів функцій (наприклад, кількість і порядок аргументів між контролером та репозиторієм) і узгодженості імен полів між шарами застосунку (DTO, Prisma-схема, відповіді API).

### 3.2 Модульне тестування (Unit)
Інструмент: **Jest** із **Babel** (`@babel/preset-env`) для підтримки ESM. Кожна функція контролерів тестувалася в ізоляції за допомогою `jest.unstable_mockModule` для підміни репозиторіїв (`organization.js`, `category.js`) та утиліти `csvParser.js`. Це дозволило перевірити логіку контролерів (валідацію, формування DTO, коди статусів) незалежно від бази даних. Утиліти `geoUtils.js` та `csvParser.js` — чисті функції, тестувалися без моків.

### 3.3 Інтеграційне тестування
Реалізовано **фейкову реалізацію Prisma Client у пам'яті** (`tests/mocks/fakePrisma.js`), яка відтворює мінімальний підмножину API (`findMany`, `findUnique`, `create`, `update`) з повною бізнес-логікою фільтрації (за статусом, категорією, географічними межами), що **точно копіює** `where`-умови, які формує реальний код репозиторіїв — включно з помилковими умовами (це дозволило виявити BUG-02 на інтеграційному рівні, а не лише в коді). Тести запускалися через `Supertest` проти повністю зібраного Express-застосунку (`src/app.js`) з підміненим лише модулем `src/db/prisma.js`.

### 3.4 API-тестування (End-to-End)
Виконано повний HTTP-цикл "запит → маршрут → middleware валідації → контролер → репозиторій → фейкова БД → відповідь" для кожного ендпоінта, описаного в наданому "Плані тестування API для MVP1". Тест-кейси з оригінального плану (`TC-01` – `TC-27`, `A-01` – `A-04`) перенесено в автоматизовані тести з тими ж ідентифікаторами для прямої трасованості.

### 3.5 Обмеження методології
Через відсутність доступу до розгорнутого сервера БД (MySQL/MariaDB) реальні запити до Prisma **не виконувалися**; натомість використано детальну фейкову реалізацію Prisma Client, що відтворює фактичну логіку `where`-умов з коду репозиторіїв. Це дозволяє з високою достовірністю перевірити бізнес-логіку та контракти API, але **не перевіряє**: коректність самої Prisma-схеми (`schema.prisma` не була надана), реальні SQL-запити, та поведінку під навантаженням реальної БД. Рекомендується додатковий цикл тестування на розгорнутому стенді (розділ 9.1).

## 6. Реєстр дефектів (Defect Log)

### BUG-01 — Критичний: масовий імпорт CSV завжди завершується помилкою 500

| Поле | Значення |
|---|---|
| Серйозність | Критична (Blocker) |
| Компонент | `src/controllers/organization.js`, функція `importCSV` |
| Виявлено на рівні | Статичний рев'ю → підтверджено модульним (TC-U32 опосередковано через мок) та інтеграційним тестуванням (TC-I31, TC-I32) |

**Опис:** У функції `importCSV` створення організації для кожного валідного рядка CSV виконується так:
```js
const org = await createOrganization({
    name: row.name.trim(),
    description: row.description?.trim() || null,
    websiteUrl: row.websiteUrl?.trim() || null,
    // TODO locations
})
```
Викликається лише з **одним** аргументом. Однак сигнатура функції в репозиторії:
```js
export async function createOrganization(newOrganization, categoryIds, locations) { ... }
```
очікує три аргументи. Усередині `categoryIds.map(...)` та `locations.map(...)` викликаються безумовно, без перевірки на `undefined`.

**Кроки відтворення:**
1. Підготувати валідний CSV-файл з колонкою `name` (наприклад, `name,description\nTest Org,Some text\n`).
2. Виконати `POST /api/organizations/import` з прикріпленим файлом.

**Очікуваний результат:** `207 Multi-Status`, `{ created: 1, errors: [] }`.

**Фактичний результат:** `500 Internal Server Error`, `{"errors":[{"field":"server","message":"Failed to create organization"}]}`. У логах сервера: `TypeError: Cannot read properties of undefined (reading 'map')` у `repositories/organization.js:17`.

**Вплив:** Функція масового імпорту **повністю непрацездатна** для будь-якого вхідного файлу, що містить хоча б один валідний рядок. Це блокує єдиний сценарій, для якого цей ендпоінт існує.

**Рекомендоване виправлення:** передавати порожні масиви за замовчуванням, наприклад:
```js
const org = await createOrganization(
    { name: row.name.trim(), description: row.description?.trim() || null, websiteUrl: row.websiteUrl?.trim() || null },
    [],   // categoryIds — CSV не містить категорій (TODO у коді)
    []    // locations — CSV не містить адрес (TODO у коді)
)
```
і додатково зробити репозиторій стійким до `undefined` (`categoryIds?.map(...) ?? []`), щоб подібний клас помилок не повторювався при майбутніх викликах з інших місць коду.

### BUG-02 — Критичний: фільтрація організацій за категорією завжди повертає порожній результат

| Поле | Значення |
|---|---|
| Серйозність | Критична (Blocker) |
| Компонент | `src/repositories/category.js`, функція `getApprovedOrganizationsByCategory` |
| Виявлено на рівні | Статичний рев'ю → підтверджено інтеграційним (TC-I29) та API-тестуванням (TC-24) |

**Опис:** Repository-функція формує Prisma-запит так:
```js
return await prisma.organization.findMany({
    where: {
        status: OrganizationStatus.approved,
        categories: {
            some: { id: Number(categoryId) },
        },
    },
    ...
});
```
Поле `categories` для організації — це зв'язок із таблицею `organization_categories` (`OrganizationCategory`), записи якої ідентифікуються через `organizationId`/`categoryId`. Умова `some: { id: Number(categoryId) }` фільтрує за полем **`id`** запису `OrganizationCategory`, а не за **`categoryId`**. Оскільки `id` зв'язкового запису ніколи не дорівнює `id` категорії (це різні автоінкрементні послідовності), умова **ніколи не виконується**, і запит завжди повертає порожній масив — незалежно від реальної наявності зв'язку.

**Порівняння з коректною реалізацією:** аналогічний фільтр у `repositories/organization.js` (функція `categoryFilter`) реалізований правильно:
```js
function categoryFilter(categoryId) {
    return categoryId === undefined ? {} : {
        categories: { some: { categoryId: Number(categoryId) } },
    }
}
```

**Кроки відтворення:**
1. Створити організацію зі статусом `approved` та зв'язком з категорією `id=3` (наприклад, через `inserts.sql`: Green Future Foundation → Environment).
2. Виконати `GET /api/categories/3/organizations`.

**Очікуваний результат:** `200 OK`, масив із принаймні однією організацією (Green Future Foundation).

**Фактичний результат:** `404 Not Found`, `{"errors":[{"field":"id","message":"No organizations found for this category"}]}` — для **будь-якої** категорії, незалежно від реальних зв'язків у БД.

**Вплив:** Ендпоінт `GET /api/categories/:id/organizations` **повністю непрацездатний**: він завжди повертає 404, навіть коли дані коректно зв'язані. Це блокує сценарій перегляду каталогу організацій за категорією — один з ключових варіантів використання продукту.

**Рекомендоване виправлення:**
```js
categories: {
    some: { categoryId: Number(categoryId) },
},
```
тобто привести фільтр до того ж формату, що вже коректно використовується в `repositories/organization.js`.

## 9. Рекомендації

### 9.1 Невідкладні (блокери релізу)
1. **Виправити BUG-01** до релізу функції масового імпорту — без цього виправлення ендпоінт `/api/organizations/import` непридатний до використання в жодному сценарії.
2. **Виправити BUG-02** до релізу функції перегляду за категоріями — `/api/categories/:id/organizations` наразі завжди повертає 404.
3. Після виправлення обох дефектів — **повторно прогнати** тести `TC-I29`, `TC-I31`, `TC-I32`, `TC-19`, `TC-24` (вони написані так, щоб одразу "почервоніти", коли дефект буде усунено, що є очікуваним сигналом для оновлення асертів на позитивні).

### 9.2 Високий пріоритет (безпека)
4. Реалізувати модуль автентифікації (JWT) та авторизації за ролями (`user`/`admin`) до того, як API стане доступним поза довіреним внутрішнім середовищем — наразі **будь-який** клієнт може змінювати статус організацій (`PUT /:id/status`) та бачити дані на модерації (`?status=pending`).
5. Додати rate-limiting на `POST /api/organizations` та `POST /api/organizations/import`, щоб запобігти зловживанню публічними ендпоінтами створення даних.

### 9.3 Середній пріоритет
6. Узгодити формат відповіді `GET /api/organizations` із продуктовою специфікацією (обгортка `{ data, pagination }` чи свідома відмова від неї з оновленням документації) — GAP-05.
7. Привести поведінку `GET /api/categories/:id/organizations` для порожнього результату до 200+`[]` замість 404, узгоджено з рештою API (де порожній список зазвичай не вважається помилкою) — GAP-03.
8. Додати явний 405 Method Not Allowed для зареєстрованих шляхів із непідтримуваним методом — GAP-04.

9. Провести цикл тестування на **реальному** розгорнутому стенді (MySQL/MariaDB через Docker Compose, наданий у проєкті) для перевірки коректності самої Prisma-схеми та реальної поведінки під навантаженням — поточний цикл використовував детальну, але штучну реалізацію БД.

11. Після впровадження автентифікації — повторно виконати розділ 1.2 та частину 1.5 плану тестування (TC-05–TC-07, TC-16) як позитивні сценарії з реальними токенами та ролями.

## 10. Висновок

Backend-частина MVP1 проєкту "Каталог організацій" демонструє **в цілому коректну архітектуру** (чітке розділення на контролери/репозиторії/маршрути/middleware, послідовна обробка помилок, валідація вхідних даних на рівні маршрутів). Базові CRUD-операції з організаціями (перегляд списку з фільтрами, перегляд за ID, створення заявки, зміна статусу) **повністю функціональні та покриті тестами без жодного відхилення**.

Водночас виявлено **два критичні дефекти**, що роблять непрацездатними дві важливі функції — масовий імпорт CSV (BUG-01) та перегляд організацій за категорією (BUG-02). Обидва дефекти мають точкове, відносно просте виправлення (додавання відсутніх аргументів та корекція назви поля у фільтрі), але є **блокуючими** для відповідного функціоналу в поточному стані.

Окремо зафіксовано, що значна частина оригінального "Плану тестування API для MVP1" (модуль авторизації, розмежування ролей `admin`/`user`) описує функціонал, який **ще не реалізований** у наданому коді — це не дефект тестування, а статус розробки, зафіксований у SRS як Roadmap.

**Рекомендація:** перед передачею MVP1 у продакшн або демонстрацію зовнішнім користувачам — усунути BUG-01 та BUG-02, оцінити ризики GAP-01/GAP-02 (публічний доступ до операцій модерації) і прийняти усвідомлене рішення щодо їх пріоритету в поточному релізі.
